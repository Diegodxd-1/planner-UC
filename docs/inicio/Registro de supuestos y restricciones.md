# 📌 REGISTRO DE SUPUESTOS Y RESTRICCIONES

### 💡 Supuestos
Los supuestos representan condiciones que se consideran verdaderas para el desarrollo del proyecto, aunque no estén completamente validadas. Estos permiten establecer un marco de referencia para el modelado del problema y la implementación del sistema.

- **Estructura Académica:** La universidad dispone de una estructura organizada en términos de cursos, docentes y aulas.
- **Responsabilidad:** Cada curso tiene asignado al menos un docente responsable.
- **Disponibilidad:** Los docentes cuentan con horarios de disponibilidad previamente definidos.
- **Exclusividad de Recursos:** Cada aula puede ser utilizada por un solo curso en un mismo horario.
- **Temporalidad:** Los cursos tienen una duración definida dentro del horario académico.
- **Usuarios Clave:** Los coordinadores académicos actuarán como principales usuarios y validadores.
- **Datos de Prueba:** Se contará con datos representativos para realizar pruebas en entorno controlado.
- **Capacidad Técnica:** Los integrantes cuentan con conocimientos en desarrollo web (Stack MERN).
- **Herramientas:** El equipo tiene acceso a GitHub, IDEs y plataformas de despliegue.

---

### 🛑 Restricciones
Las restricciones representan condiciones que limitan o condicionan el desarrollo del proyecto, influyendo directamente en su alcance y diseño.

- **Tiempo:** El proyecto debe desarrollarse dentro de un periodo estricto de **16 semanas**.
- **Alcance:** El sistema se desarrollará como un **Producto Mínimo Viable (PMV)**.
- **Tecnología:** Uso obligatorio del stack **MERN** (MongoDB, Express, React, Node.js).
- **Colaboración:** El desarrollo debe ser en equipo con control de versiones formal (Git).
- **Metodología:** Gestión bajo el marco ágil **Scrum** con iteraciones cortas.
- **Repositorio:** Código fuente almacenado en GitHub con evidencia de Pull Requests.
- **Arquitectura:** Separación obligatoria de Frontend y Backend.
- **Documentación:** Formato Markdown accesible desde el README principal.
- **Accesibilidad:** Sistema accesible desde navegadores modernos sin instalación.
- **Validación:** No se contará con integración a sistemas reales; validación en entorno controlado.

---

### ⚖️ Validación de las Restricciones
Las restricciones del proyecto han sido definidas considerando factores técnicos, académicos y organizacionales. Se ha tomado en cuenta el tiempo disponible (**16 semanas**), la experiencia del equipo y los lineamientos del curso. Estas restricciones han sido contrastadas con la naturaleza del problema para asegurar su viabilidad. Por ejemplo, el enfoque PMV permite ajustar el alcance al cronograma académico.

---

### 🔗 Coherencia entre Supuestos y Restricciones
Se evidencia una alta coherencia entre ambos elementos:
1.  El supuesto de **conocimientos técnicos** es consistente con la restricción del **Stack MERN**.
2.  La **disponibilidad de datos** (supuesto) se alinea con la **validación controlada** (restricción).
3.  El desarrollo como **PMV** es la respuesta lógica a la restricción de **tiempo limitado**.

> [!NOTE]
> En conjunto, no se identifican contradicciones, lo que garantiza la consistencia del planteamiento estratégico del proyecto.
